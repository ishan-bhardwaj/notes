# Structured API - DataFrames, Datasets, Spark SQL

- Spark's high-level abstraction over the low-level RDD engine
- Datasets, DataFrames, Spark SQL - all compile down to the same logical/physical plan and produce identical bytecode at execution time
- No performance difference between writing `df.groupBy(...)` vs `spark.sql("SELECT ... GROUP BY")`
  - they go through the exact same Catalyst pipeline

## Datasets

- Strongly typed distributed collection -
    - Scala - `Dataset[T]`
    - Java - `Dataset<T>`

- Example -
    ```
    case class Person(
        name: String,
        age: String
    )

    val personDF = spark.read.parquet("/data/persons.parquet")
    val persons = personDF.as[Flight]
    ```

- Actions like `collect()` or `take()` on a Dataset returns typed objects, not generic `Row` -
    ```
    val result: Array[Person] = persons.take(5)
    ```

### Encoders

- Dataset uses generated bytecode encoders (`Encoder[T]`) to convert JVM objects to Spark's internal Tungsten binary format without going through Java serialization - extremely fast
- Dataset types must be Scala case classes or Java JavaBean-style classes - 
    - Spark must be able to inspect the type structure at compile time to generate the encoder

## DataFrames

- Spark’s main abstraction for working with structured/tabular data
- A DataFrame is a dataset organized into named columns -
    - Scala - `DataFrame` = `Dataset[Row]`
    - Java - `DataFrame` = `Dataset<Row>`
- Immutable - every operation creates a new DataFrame
- DataFrame consists of `Row`s and schema

## Dataset vs DataFrame

| Aspect                  | `Dataset[T]`                                                        | `DataFrame` (`Dataset[Row]`)                           |
| ----------------------- | ------------------------------------------------------------------- | ---------------------------------------------------- |
| Definition              | Strongly typed distributed collection of JVM objects                | Untyped distributed table with named columns         |
| Type Safety             | Compile-time type safety                                            | Runtime-only type checking                           |
| Schema                  | Derived from case classes / JVM types                               | Explicit or inferred schema                          |
| Column Access           | `.name`, `.age` (typed fields)                                      | `col("name")`, `$"name"`                             |
| Internal Representation | JVM objects + Encoders                                              | Tungsten binary row format                           |
| Serialization           | Encoders (generated bytecode)                                       | UnsafeRow / Tungsten format                          |
| Compile-Time Validation | Yes                                                                 | No                                                   |
| Performance             | Slight overhead from object encoding/decoding                       | Generally faster for SQL-style workloads             |
| Optimizer Support       | Catalyst optimized                                                  | Catalyst optimized                                   |
| Best For                | Complex domain/business logic in Scala/Java                         | ETL, analytics, SQL-style transformations            |
| Language Support        | Scala & Java only                                                   | Scala, Java, Python, R, SQL                          |
| Functional Operations   | Strong typed functional APIs (`map`, `flatMap`, typed aggregations) | Mostly relational APIs (`select`, `groupBy`, `join`) |
| Null Handling           | JVM object semantics                                                | SQL null semantics                                   |
| Recommended Usage       | Type-heavy application logic                                        | Data engineering and analytical workloads            |

## Schema

- A schema is a `StructType` - an ordered list of `StructField(name, dataType, nullable)`
- Schema is the contract between Spark and the data - all operations are validated against it
- Two ways to get a schema -
    - __Schema-on-read (inferSchema)__ - 
        - Spark samples the data and guesses types
        - Slow (extra scan), unreliable on edge cases, never use in production
    - __Explicit schema__ - 
        - Define it yourself
        - Always do this in production for speed, stability, and predictable types
- Schema is attached to the DataFrame, not to individual Rows - 
    - A `Row` object is unstructured (just an array of `Any`)
    - The schema lives on the DataFrame that wraps it

## Partitions

- A DataFrame is physically split into __partitions__ - 
    - Each partition is a contiguous chunk of rows sitting on one node
- Partition count = degree of parallelism - one task processes one partition
- Spark maintains a __lineage__ (the DAG of transformations) for every DataFrame - 
    - If a partition is lost (executor dies), Spark recomputes it by replaying the lineage from the source
    - No data replication needed
- Partitions are an internal detail - 
    - The DataFrame API has no concept of individual partitions from the user's perspective
    - Unlike RDD where you control them explicitly

## Transformations

- Describe how to produce a new DataFrame from existing ones
- Two types -
    - __Narrow Transformations__ - 
        - Each input partition maps to exactly one output partition
        - No data movement
        -  Spark performs _pipelining_ automatically - multiple narrow transformations are combined together
        - Eg - `filter`, `select`, `withColumn`, `map`, `union`
    - __Wide Transformations__ - 
        - One input partition contributes to _many_ output partition
        - Causes shuffles - redistribution of data across cluster
        - Eg - `groupBy`, `join`, `sort`, `distinct`, `repartition`

- __Lazily evaluated__ - calling `filter()`, `select()`, `join()` etc. builds the logical plan but executes nothing

### Shuffle

- Wide transformations force a shuffle - map-side tasks write partitioned output to local disk
- Reduce-side tasks fetch their slice from all map outputs over the network
- `spark.sql.shuffle.partitions` (default $200$) controls how many partitions the shuffle output produces -
    - $200$ is wrong for almost every job
- Adaptive Query Execution (AQE) can dynamically coalesce shuffle partitions at runtime -
    - Largely fixes the "$200$ is wrong" problem in Spark 3.x

## Actions

- Actions trigger the actual execution - they materialize the lazy plan into a Spark job
- One action = one Spark job submitted to the scheduler
- Actions that return data to the driver - `collect()`, `take(n)`, `first()`, `count()`
- Actions that write to storage - `df.write.parquet(...)`, `df.write.jdbc(...)`
- `collect()` pulls all rows to the Driver - fine for small results, OOM risk for large DataFrames

## SparkSession

- Entry point to all Spark functionality (2.0+).
- Creating SparkSession -
    ```
    from pyspark.sql import SparkSession

    spark = SparkSession \
        .builder \
        .appName("Python Spark SQL basic example") \
        .config("spark.some.config.option", "some-value") \
        .getOrCreate()
    ```

> [!NOTE]
> SparkSession in 2.0+ provides built-in support for Hive features - HiveQL queries, Hive UDFs, reading Hive tables. No existing Hive setup is needed to use these features.

## DataFrameReader

- API for reading external data into Spark - accessed via `spark.read`
- At read time, Spark does not load data into memory - 
    - It builds a __scan plan__ (which files to read, which columns to project, which predicates to push down to the source)
- Actual data is only read when an action triggers execution
- Generic API - `spark.read.format(...).option(...).load(path)`
- Convenience APIs - `spark.read.parquet(path)`, `.csv(path)`, `.json(path)` - same thing internally
- Read modes for malformed records (JSON/CSV) -
    - `permissive` (default) - puts nulls for bad fields, keeps the row
    - `dropMalformed` - silently drops bad rows
    - `failFast` - throws exception immediately on first bad record

## DataFrameWriter

- API for writing DataFrames to storage - accessed via `df.write`
- Save modes -
    - `overwrite` - replace existing data
    - `append` - add to existing data
    - `ignore` - silently skip if target already exists
    - `errorIfExists` (default) - throw if target already exists
- Partition pruning at write - `.partitionBy("date", "region")` 
    - Physically organizes files into a Hive-style directory hierarchy
    - Downstream reads with a filter on those columns and skip irrelevant directories entirely (partition pruning)
- Bucketing at write - `.bucketBy(n, "key").sortBy("key")` 
    - Pre-partitions and pre-sorts data
    - Joins between two bucketed tables on the same key with the same bucket count require _no shuffle at all_ - 
        - Most powerful write-time optimization for repeatedly-joined large tables

## Data Formats

- __Parquet__ - default format for Spark; columnar, splittable, schema-embedded, highly compressed;
  supports predicate pushdown (Spark pushes filters into the file reader - rows never enter JVM
  memory) and column pruning (only requested columns read off disk)
- __JSON__ - row-oriented, human-readable, verbose; inferSchema is especially unreliable here
  (types change based on which rows are sampled); `dateFormat` option matters when parsing dates
- __CSV__ - flat, no schema, no nesting; always specify `header`, `sep`, `nullValue` explicitly;
  all columns default to `StringType` without a schema
- __JDBC__ - reads from relational DBs; parallelism controlled by `numPartitions`, `partitionColumn`,
  `lowerBound`, `upperBound` - without these, Spark uses a single partition (single DB connection,
  no parallelism); predicate pushdown supported (WHERE clause sent to DB)
- __ORC__ - alternative columnar format; slightly better compression than Parquet in some cases;
  default in Hive ecosystems

## Spark SQL

- Register any DataFrame as a SQL-queryable view -
    - `df.createOrReplaceTempView("name")` - session-scoped - gone when SparkSession ends
    - `df.createGlobalTempView("name")` - app-scoped - shared across sessions within the same application
- Compile to identical logical plans as the DataFrame API - zero performance difference
- Global temp views live in the reserved `global_temp` database - 
    - Must use the qualified name `global_temp.view_name` to reference them

## Interoperating with RDDs

- Method 1 - __Inferring Schema via Reflection__ -
  - Works when schema known at compile time
  - Rows are constructed by passing a list of key/value pairs as kwargs to the `Row` class
  - The keys of this list define the column names of the table, and the types are inferred by sampling the whole dataset
  - Eg -
    ```
    from pyspark.sql import Row

    lines = sc.textFile("examples/src/main/resources/people.txt")
    parts = lines.map(lambda l: l.split(","))
    people = parts.map(lambda p: Row(name=p[0], age=int(p[1])))

    schemaPeople = spark.createDataFrame(people)
    schemaPeople.createOrReplaceTempView("people")

    teenagers = spark.sql("SELECT name FROM people WHERE age >= 13 AND age <= 19")
    teenNames = teenagers.rdd.map(lambda p: "Name: " + p.name).collect()
    ```

- Method 2 - __Programmatically Specifying Schema__ -
  - Used when a dictionary of kwargs cannot be defined ahead of time
  - Steps -
    - Create an RDD of tuples or lists from the original RDD
    - Create the schema represented by a `StructType` matching the structure of tuples or lists in the RDD created in the step 1
    - Apply the schema to the RDD via `createDataFrame` method provided by `SparkSession`
  - Eg -
    ```
    from pyspark.sql.types import StringType, StructType, StructField

    lines = sc.textFile("examples/src/main/resources/people.txt")
    people = lines.map(lambda l: tuple(l.split(",")))

    schemaString = "name age"
    fields = [StructField(field_name, StringType(), True) for field_name in schemaString.split()]
    schema = StructType(fields)

    schemaPeople = spark.createDataFrame(people, schema)
    schemaPeople.createOrReplaceTempView("people")
    results = spark.sql("SELECT name FROM people")
    results.show()
    ```
