# Garbage Collection

## GC Overview

- GC finds objects no longer reachable from __GC roots__ (thread stacks, system classes) and frees their memory
- Reference counting is insufficient — circular references and disconnected linked lists would never be freed
- After freeing objects, heap must be __compacted__ to eliminate fragmentation; without compaction, large allocations fail even when total free space is sufficient
- __Mutator threads__ — application threads that create/modify objects
- __Stop-the-world (STW) pause__ — all application threads halted while GC works; minimising these is the primary GC tuning goal

---

## Generational Collectors

### Young Generation (Eden + Survivor Spaces)

- Most objects die young — allocating everything in a small region (young gen) makes collection fast
- Allocation happens in __eden__; when young gen fills, a __minor GC__ (young GC) runs
- Live objects moved to a survivor space or promoted to old gen; eden is left completely empty → automatic compaction
- All current algorithms use STW for minor GC, but pauses are short because the region is small

### Old (Tenured) Generation

- Objects that survive enough young GCs are promoted here
- When old gen fills, a __full GC__ is triggered
- Simple collectors stop all threads, find garbage, free memory, and compact (long pause)
- __Concurrent collectors__ find garbage while application threads run (shorter pauses, more CPU cost)

### Trade-offs

- Larger young gen → longer minor GC pauses, fewer promotions to old gen, fewer full GCs
- Smaller young gen → shorter but more frequent minor GCs, more promotions, more full GCs
- Concurrent collection avoids long full GC pauses but consumes CPU for background threads

## GC Algorithms

| Algorithm | JDK 8 | JDK 11 | JDK 12 | Notes |
|---|---|---|---|---|
| Serial GC | Supported | Supported | Supported | Single-threaded; default on 1 CPU |
| Throughput (Parallel) GC | Supported | Supported | Supported | Default in JDK 8 on ≥2 CPUs |
| G1 GC | Supported | Supported | Supported | Default in JDK 11+ on ≥2 CPUs |
| CMS | Supported | Deprecated | Deprecated | No heap compaction; avoid in new code |
| ZGC | — | Experimental | Experimental | |
| Shenandoah | — | Experimental | Experimental | Not in Oracle builds |
| Epsilon GC | — | Experimental | Experimental | No-op collector |

### Serial GC (`-XX:+UseSerialGC`)

- Single thread for all GC; STW for minor and full GC; compacts old gen on full GC
- Default on: single-CPU machines, 32-bit Windows JVMs, Docker containers limited to 1 CPU
- Surprisingly competitive when only 1 CPU is available — avoids multi-thread coordination overhead

### Throughput (Parallel) GC (`-XX:+UseParallelGC`)

- Multiple threads for young and old gen collection
- STW for both minor and full GC; compacts old gen on full GC
- Default in JDK 8 on 64-bit machines with ≥2 CPUs
- Maximises total throughput; full GC pauses can be long
- `-XX:+UseParallelOldGC` is obsolete (parallel old gen is always included)

### G1 GC (`-XX:+UseG1GC`)

- Heap divided into equal-sized regions; some designated young, rest old
- Young gen collected STW with multiple threads (same as throughput collector for minor GC)
- Old gen collected by __background threads__ concurrently with application threads
- Compacts heap incrementally by copying between regions during normal operation
- Default in JDK 11+ on 64-bit machines with ≥2 CPUs
- Trade-off: background threads compete with application threads for CPU

### CMS (`-XX:+UseConcMarkSweepGC`)

- First concurrent collector; deprecated in JDK 11
- No heap compaction — heap eventually fragments, forcing a STW compaction (defeating the purpose)
- Superseded by G1 GC; avoid in new code

## Choosing a GC Algorithm

### Single CPU / Container with 1 CPU

| Scenario | Best Choice |
|---|---|
| CPU-bound batch job | Serial GC |
| REST server, response time matters | G1 GC (avoids full GC outliers) |
| REST server, average time matters | Serial GC (simpler = faster minor GC) |
| No full GCs occurring | All roughly equivalent |

Key insight: on 1 CPU, G1 GC background threads steal CPU from application threads even when "idle"

### Single Hyper-threaded CPU (appears as 2 CPUs to JVM)

- JVM defaults to throughput (JDK 8) or G1 GC (JDK 11), NOT serial
- For CPU-bound batch jobs: serial GC still wins (HT cores share execution units)
- For response-time-sensitive work: G1 GC still better for 99th% outliers

### Multi-CPU Machines (≥2 real cores)

| Scenario | Best Choice |
|---|---|
| General case | G1 GC |
| CPU-bound with few full GCs | Throughput GC |
| Old generation nearly always full | Throughput GC |
| G1 GC background threads starved for CPU | Throughput GC |
| Response time / 99th% latency critical | G1 GC |

CPU usage pattern difference:
- Throughput GC: sawtooth — 100% CPU spikes during GC, lower between
- G1 GC: stepped — base CPU + extra when background threads run; no 100% spikes (except very brief minor GC)

### `System.gc()`

- Always triggers a full GC regardless of the algorithm in use
- Almost always a bad idea; only acceptable in benchmarks or heap analysis
- RMI calls `System.gc()` hourly by default: tune with `-Dsun.rmi.dgc.server.gcInterval=N` (ms)
- Disable entirely with `-XX:+DisableExplicitGC`

## Basic GC Tuning

### Heap Sizing

Flags:
- `-XmsN` — initial heap size
- `-XmxN` — maximum heap size

Default values (Linux 64-bit):
- Initial: `min(512 MB, physicalRAM / 64)`
- Maximum: `min(32 GB, physicalRAM / 4)`

> [!NOTE]
> JDK 8 before update 192 uses host machine RAM for defaults even inside Docker containers. JDK 8u192+ and JDK 11+ respect container memory limits.

Rules:
- Never exceed physical RAM (sum of all JVM heaps + OS overhead)
- Leave ≥ 1 GB for OS and other processes
- Good rule of thumb: heap should be ~30% occupied after a full GC at steady state
- Setting `-Xms` = `-Xmx` eliminates heap resize overhead (slightly more efficient GC)
- JVM grows the heap as needed up to Xmx; a large Xmx does not waste committed memory

### Generation Sizing

Flags:
- `-XX:NewRatio=N` (default 2) — `youngGenSize = heapSize / (1 + NewRatio)` → ~33% young at default
- `-XX:NewSize=N` — initial young gen size (overrides NewRatio)
- `-XX:MaxNewSize=N` — maximum young gen size
- `-XmnN` — shorthand for `NewSize = MaxNewSize = N` (fixed young gen size)

When to tune:
- Fixed heap (`-Xms` = `-Xmx`): use `-Xmn` for a fixed young gen
- Dynamic heap: tune `NewRatio` instead

### Adaptive Sizing

- JVM automatically resizes young gen, old gen, and survivor spaces between GC cycles
- Goal: meet GC performance targets (pause time, throughput)
- Enabled by default: `-XX:+UseAdaptiveSizePolicy`
- Disable if heap is perfectly tuned and phases are stable: `-XX:-UseAdaptiveSizePolicy`
- Debug resizing decisions: `-XX:+PrintAdaptiveSizePolicy`
- Effectively disabled when `-Xms` = `-Xmx` and `-Xmn` is set

### Metaspace Sizing

- Holds class __metadata__ (not class objects or reflection objects — those are on the main heap)
- Default initial: ~20 MB (64-bit); default maximum: unlimited
- Flags: `-XX:MetaspaceSize=N`, `-XX:MaxMetaspaceSize=N`
- Resizing requires a full GC — increase initial size if many full GCs occur at startup
- Servers often set initial metaspace to 128–256 MB
- Classloader leaks can fill metaspace; diagnose with heap dumps or `jmap -clstats`

### Parallel GC Thread Count

Flag: `-XX:ParallelGCThreads=N`

Default formula:
- ≤8 CPUs: one thread per CPU
- \>8 CPUs: `8 + ((N - 8) * 5 / 8)`

Affects: young gen collection (parallel/G1), old gen collection (parallel), STW phases of G1

When to reduce:
- Small heap (1 GB) on large machine — fewer threads more efficient
- Multiple JVMs on one machine — total threads across all JVMs becomes too many
- Docker containers (JDK 11 uses container CPU limit; older JDK 8 uses host CPU count)

Rule of thumb for multiple JVMs: `ParallelGCThreads = numCPUs / numJVMs`

## GC Logging and Tools

### JDK 8 GC Logging Flags

```bash
-Xloggc:gc.log                    # write to file (enables basic log)
-XX:+PrintGCDetails               # detailed log (recommended)
-XX:+PrintGCTimeStamps            # relative timestamps
-XX:+PrintGCDateStamps            # absolute date/time stamps
-XX:+UseGCLogFileRotation         # enable rotation
-XX:NumberOfGCLogFiles=8          # keep 8 files
-XX:GCLogFileSize=8m              # rotate at 8 MB
```

Recommended production config:

```bash
-Xloggc:gc.log -XX:+PrintGCDetails -XX:+PrintGCTimeStamps \
-XX:+UseGCLogFileRotation -XX:NumberOfGCLogFiles=8 -XX:GCLogFileSize=8m
```

### JDK 11 GC Logging (Unified Logging)

```bash
-Xlog:gc*:file=gc.log:time:filecount=7,filesize=8M
```

Format: `-Xlog:<what>:<output>:<decorators>:<output-options>`

- `gc*` — all GC modules at info level
- `gc*,gc+age=debug` — all GC at info + tenuring at debug
- Decorators: `time`, `uptime`, `pid`, `tid`, etc.

> [!NOTE]
> Log rotation numbering differs between JDK 8 and 11. JDK 11 uses one extra active file; specify `filecount=7` to keep 8 total files.

### jstat

```bash
jstat -gcutil <pid> 1000     # GC utilisation % every 1s
jstat -gc <pid> 1000         # GC sizes in KB every 1s
jstat -options               # list all available options
```

Key columns in `-gcutil`:
- `S0`, `S1` — survivor space utilisation %
- `E` — eden utilisation %
- `O` — old gen utilisation %
- `P`/`M` — permanent gen / metaspace utilisation %
- `YGC` / `YGCT` — young GC count / total time
- `FGC` / `FGCT` — full GC count / total time
- `GCT` — total GC time

### Other Tools

- `jconsole` / `jvisualvm` — real-time heap graphs (Memory panel)
- `jcmd <pid> GC.run` — force a full GC
- GCeasy (web), jClarity Censum — log parsing and visualisation

# Garbage Collection Algorithms

## Throughput (Parallel) GC

### Young Collection

- Minor GC triggered when eden fills up
- All live objects moved from eden → survivor space or old gen; remainder discarded
- Eden automatically compacted (emptied) after every young collection

JDK 8 log:
```
17.806: [GC (Allocation Failure) [PSYoungGen: 227983K->14463K(264128K)]
         280122K->66610K(613696K), 0.0169320 secs]
         [Times: user=0.05 sys=0.00, real=0.02 secs]
```
- `PSYoungGen: before→after(capacity)` — young gen occupancy
- Second pair — entire heap before→after(capacity)
- `user` > `real` — parallel GC threads (4 threads × CPU time = user time)

### Full GC

- Young gen completely cleared (survivor spaces too)
- Old gen compacted: live objects moved to front, rest freed
- Metaspace unchanged (unless it triggered the full GC)

JDK 8 log:
```
64.546: [Full GC (Ergonomics) [PSYoungGen: 15808K->0K(339456K)]
          [ParOldGen: 457753K->392528K(554432K)] 473561K->392528K(893888K)
          [Metaspace: 56728K->56728K(115392K)], 1.3367080 secs]
```

### Throughput Collector Tuning

Two key flags for adaptive sizing:
- `-XX:MaxGCPauseMillis=N` — max acceptable GC pause; takes precedence over throughput goal; unset by default
    - Setting too low → very small old gen → extremely frequent full GCs
- `-XX:GCTimeRatio=N` (default 99) — throughput goal: `appThreadTime% = N / (N + 1)`
    - Default 99 → 99% app time, 1% GC time
    - To get 95% app time: `GCTimeRatio = 0.95 / (1 - 0.95) = 19`

Sizing process (with adaptive sizing):
1. JVM adjusts gen sizes to meet `MaxGCPauseMillis` first
2. Then expands heap until `GCTimeRatio` goal is met
3. Then shrinks heap to smallest size that meets both goals

Static sizing approach:
- Set `-Xms = -Xmx` to eliminate resize overhead
- Experiment with heap sizes to find the knee of the throughput curve
- Beyond ~4,500 MB in example: longer GC cycles offset the benefit of less frequent GC

## G1 GC

### Region Structure

- Heap divided into ~2,048 equal-sized regions (1–32 MB each, always a power of 2)
- Regions assigned to young or old gen dynamically; non-contiguous
- `Humongous` regions — special old-gen regions for very large objects

### Four Logical Operations

1. **Young collection** — STW, evacuates all eden regions
2. **Concurrent marking cycle** — background threads find old-gen garbage
3. **Mixed GC** — STW young collection + cleans some marked old-gen regions
4. **Full GC** — only on failure; single-threaded in JDK 8, parallel in JDK 11

### Young Collection

- Triggered when eden fills
- All live objects from eden → survivor regions or old-gen regions
- Eden and surplus survivor regions freed

JDK 8 log:
```
23.430: [GC pause (young), 0.23094400 secs]
   [Eden: 1286M(1286M)->0B(1212M) Survivors: 78M->152M Heap: 1454M(4096M)->242M(4096M)]
```

### Concurrent Marking Cycle

Phases:

| Phase | STW? | Description |
|---|---|---|
| Initial-mark (JDK8) / Concurrent Start (JDK11) | Yes (piggybacks on young GC) | Scans GC roots; triggers background cycle |
| Root region scan | No | Scans survivor regions; cannot be interrupted by young GC |
| Concurrent mark | No | Marks live objects across old gen |
| Remark | Yes (short) | Finalises mark; processes reference objects |
| Cleanup | Yes (short) then No | Identifies mostly-garbage regions; concurrent cleanup follows |

> [!NOTE]
> If a young GC is needed while root region scanning runs, the young GC must wait for root scanning to finish. This causes a longer-than-normal young GC pause. Tuning to prevent this is important.

Concurrent start log (JDK 8):
```
50.541: [GC pause (G1 Evacuation pause) (young) (initial-mark), 0.27767100 secs]
```

Concurrent mark log:
```
111.382: [GC concurrent-mark-start]
120.905: [GC concurrent-mark-end, 9.5225160 sec]
```

### Mixed GC

- Triggered after concurrent cycle completes
- Collects all young regions + some marked old-gen regions (those ≥85% garbage)
- Series of mixed GCs until marked regions are cleaned; then resumes normal young GCs

JDK 8 log:
```
79.826: [GC pause (mixed), 0.26161600 secs]
   [Eden: 1222M(1222M)->0B(1220M) Survivors: 142M->144M Heap: 3200M(4096M)->1964M(4096M)]
```

### Failure Modes (trigger full GC)

| Failure | Cause | Symptom in log |
|---|---|---|
| Concurrent mode failure | Old gen fills before marking cycle completes | `GC concurrent-mark-abort` + `Full GC` |
| Promotion failure | Mixed GC doesn't free space fast enough; young GC can't promote | `Full GC` after `(mixed)` |
| Evacuation failure / to-space overflow | No room in survivors or old gen for live objects | `(to-space exhausted)` in young GC |
| Humongous allocation failure | No contiguous regions for large object | `Pause Full (G1 Humongous Allocation)` |
| Metadata GC threshold (JDK 8) | Metaspace full | Young GC + Full GC labelled `(Metadata GC Threshold)` |

Full GC in JDK 8 is **single-threaded** — very long pauses. JDK 11 full GC is parallel — much shorter. This is a key reason to prefer JDK 11 with G1.

### G1 GC Tuning

Primary flag: `-XX:MaxGCPauseMillis=N` (default 200ms)
- G1 adjusts young gen size, heap size, background activity, and mixed GC aggressiveness to meet this target
- Reducing this value shrinks young gen (more frequent young GCs) and reduces old gen regions per mixed cycle (higher concurrent failure risk)

**Increasing old gen space:**
- Increase overall heap size (`-Xmx`)
- Adjust young-to-old ratio via `-XX:NewRatio=N`

**More concurrent threads:**
- `-XX:ConcGCThreads=N` — background marking threads (default: `(ParallelGCThreads + 2) / 4`)
- `-XX:ParallelGCThreads=N` — STW GC threads

**Start background marking sooner:**
- `-XX:InitiatingHeapOccupancyPercent=N` (default 45) — % of entire heap that triggers a concurrent cycle
- Set too high → concurrent mode failures
- Set too low → excessive background pauses and wasted CPU
- Check heap occupancy after a cycle to calibrate; set above that value

**More old gen regions per mixed GC:**
- `-XX:G1MixedGCCountTarget=N` (default 8) — max number of mixed GC cycles to process marked regions
- Decrease → fewer mixed GC cycles, more work per cycle, longer pauses, fewer concurrent failures
- Increase → more mixed cycles, less work per cycle, shorter pauses, higher concurrent failure risk

## CMS (Deprecated)

> [!NOTE]
> CMS is deprecated in JDK 11. The primary flaw: no concurrent compaction. Heap fragmentation eventually forces a single-threaded full GC, which can take 10–30+ seconds on large heaps. G1 GC is preferred.

### Phases

| Phase | STW? | Description |
|---|---|---|
| Initial-mark | Yes | Find all GC roots in old gen |
| Concurrent-mark | No | Mark live objects from roots |
| Concurrent-preclean | No | Pre-process cards changed during marking |
| Abortable-preclean | No | Wait until young gen is ~50% full to avoid back-to-back pauses |
| Remark | Yes (short) | Final marking sweep |
| Concurrent-sweep | No | Free unmarked objects |
| Concurrent-reset | No | Reset CMS data structures |

Failure modes:
- __Concurrent mode failure__ — old gen fills before sweep finishes; triggers single-threaded full GC + compaction (6+ seconds)
- __Promotion failure__ — old gen fragmented; can't hold promoted objects; triggers single-threaded full GC + compaction (28+ seconds)
- __Metaspace full__ — triggers full GC

### CMS Tuning

**Prevent concurrent mode failures:**

Start background threads sooner:
```bash
-XX:CMSInitiatingOccupancyFraction=N   # default 70; start cycle at N% old gen occupancy
-XX:+UseCMSInitiatingOccupancyOnly     # use only this threshold, not adaptive algorithm
```
- Find the right value: look at `CMS-initial-mark` before a failure; set below that percentage
- Don't set too low — wastes CPU and adds more short pauses

More background threads:
```bash
-XX:ConcGCThreads=N   # default: (3 + ParallelGCThreads) / 4
```

CMS and adaptive sizing: CMS never resizes young gen unless a full GC occurs (unlike G1). Set initial heap large enough to avoid startup failures.

## Advanced Tunings

### Survivor Spaces and Tenuring

Survivor spaces allow short-lived objects extra chances to be collected in the young gen before promotion.

Each young collection alternates between two survivor spaces:
- Eden → survivor space 0 → survivor space 1 → ... → old gen (after `MaxTenuringThreshold` cycles)
- Survivor space overflow → objects promoted directly to old gen (bad — fills old gen with short-lived objects)

Key flags:
- `-XX:InitialSurvivorRatio=N` (default 8) — initial survivor size = `newSize / (N + 2)`
- `-XX:MinSurvivorRatio=N` (default 3) — maximum survivor size = `newSize / (N + 2)` → default max = 20% of young gen
- `-XX:TargetSurvivorRatio=N` (default 50) — survivor space is resized to be N% full after GC
- `-XX:InitialTenuringThreshold=N` (default 7 for throughput/G1, 6 for CMS)
- `-XX:MaxTenuringThreshold=N` (default 15 for throughput/G1, 6 for CMS)

View tenuring distribution:
- JDK 8: `-XX:+PrintTenuringDistribution`
- JDK 11: `age*=debug` or `age*=trace` in Xlog config

Signs of overflow:
- `new threshold 1 (max 15)` in log — JVM promoting almost everything immediately
- G1 GC log shows: `age 1: X bytes, age 2: Y bytes, total > desired survivor size`

Remedies (in order of preference):
1. Increase overall heap size so survivor spaces can grow with young gen
2. Decrease `SurvivorRatio` (increases survivor share of young gen) — reduces eden; may increase minor GC frequency
3. Increase `MaxTenuringThreshold` — only if objects genuinely die shortly after threshold

Special flags:
- `-XX:+AlwaysTenure` — always promote to old gen after first young GC
- `-XX:+NeverTenure` — never promote (objects stay in survivor forever unless overflow)

### Thread-Local Allocation Buffers (TLABs)

- Each thread has a private TLAB carved from eden; eliminates synchronisation for allocation
- Objects larger than TLAB → allocated directly from heap (requires sync; slower)
- TLABs enabled by default: `-XX:+UseTLAB`

When allocation outside TLAB is high → either reduce object size or tune TLAB:
- `-XX:TLABSize=N` (default 0 = dynamic) — set fixed initial TLAB size
- `-XX:-ResizeTLAB` — disable dynamic resizing
- `-XX:TLABWasteTargetPercent=N` (default 1) — % of TLAB that can be wasted before retiring it
- `-XX:MinTLABSize=N` (default 2 KB) — minimum TLAB size

Diagnosis:
- JFR: TLAB allocation panel shows count and size of out-of-TLAB allocations
- JDK 8: `-XX:+PrintTLAB` — prints per-thread TLAB stats at each young GC
- JDK 11: `tlab*=trace` or `tlab*=debug` in Xlog config

### Humongous Objects (G1 GC)

- Humongous = object larger than half the G1 region size (e.g., > 512 KB if region = 1 MB)
- Allocated directly in old gen as contiguous regions; cannot be collected in young GC
- Freed during cleanup phase of concurrent cycle (not mixed GC)
- Was previously a major source of full GCs; improved in JDK 8u60 and all JDK 11 builds

### G1 Region Sizing

| Heap Size | Default Region Size |
|---|---|
| < 4 GB | 1 MB |
| 4–8 GB | 2 MB |
| 8–16 GB | 4 MB |
| 16–32 GB | 8 MB |
| 32–64 GB | 16 MB |
| > 64 GB | 32 MB |

Override: `-XX:G1HeapRegionSize=N` (must be power of 2; 1 MB–32 MB)

When to change:
- Very wide heap range (e.g., `-Xms2G -Xmx32G`) → many tiny regions → increase region size toward 2,048 total regions at expected heap size
- Applications allocating objects > half the region size → increase region size so objects fit within one region

## Experimental Collectors

### ZGC (`-XX:+UseZGC`)

- Available in JDK 11 (experimental); Oracle and AdoptOpenJDK builds
- Concurrent compaction — heap is compacted while application threads run
- Non-generational — single heap; no young/old split
- Target pause time: < 10 ms
- Application threads use barriers on object access; slight throughput overhead (~5–10%)

### Shenandoah GC (`-XX:+UseShenandoahGC`)

- Available in JDK 12+ (experimental); AdoptOpenJDK only (not Oracle builds); backported to JDK 8/11
- Also concurrent compaction with similar design to ZGC
- Non-generational

### Latency Comparison (REST server, fixed load, 500 OPS)

| Collector | Avg | 90th% | 99th% | Max |
|---|---|---|---|---|
| Throughput GC | 13 ms | 60 ms | 160 ms | 265 ms |
| G1 GC | 5 ms | 10 ms | 35 ms | 87 ms |
| ZGC | 1 ms | 5 ms | 5 ms | 20 ms |
| Shenandoah | 1 ms | 5 ms | 5 ms | 22 ms |

Enable experimental collectors:
```bash
-XX:+UnlockExperimentalVMOptions -XX:+UseZGC
-XX:+UnlockExperimentalVMOptions -XX:+UseShenandoahGC
```

### Epsilon GC (`-XX:+UseEpsilonGC`)

- No-op collector — objects are never freed; OOM when heap fills
- Uses less memory (no generational overhead) and has no GC overhead
- Useful only for: very short-lived programs, or programs carefully written to never allocate after startup

```bash
-XX:+UnlockExperimentalVMOptions -XX:+UseEpsilonGC
```

## GC Tuning Decision Tree

1. **Can the application tolerate full GC pauses?**
    - No → use G1 GC

2. **Are default settings sufficient?**
    - Check GC log — if spending ≤ 3% of time in GC, tuning unlikely to help much

3. **Are pause times close to your goal?**
    - Yes → adjust `-XX:MaxGCPauseMillis`
    - No, too long but throughput OK → reduce young gen size (more but shorter pauses)

4. **Is throughput lagging despite short pauses?**
    - Increase heap size (larger young gen → fewer minor GCs)

5. **Concurrent collector showing concurrent mode failures?**
    - More CPU available → increase ConcGCThreads
    - Increase heap size
    - Start background cycle sooner: lower `InitiatingHeapOccupancyPercent`
    - G1: reduce `G1MixedGCCountTarget`

6. **Concurrent collector showing promotion / evacuation failures?**
    - G1: start background cycle sooner, reduce `G1MixedGCCountTarget`, increase heap

7. **Full GCs dominating?**
    - Increase heap
    - Switch to concurrent collector (G1 or experimental)
