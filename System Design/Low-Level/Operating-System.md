## Operating Systems

## Terminology

| Term | Definition |
|------|------------|
| Operating system | Software and files installed on a system to boot and execute programs — includes kernel, admin tools, and system libraries |
| Kernel | Program that manages the system — hardware devices, memory, CPU scheduling — runs in privileged kernel mode |
| Process | OS abstraction and environment for executing a program — runs in user mode |
| Thread | Executable context scheduled to run on a CPU — has stack, registers, instruction pointer |
| Task | Linux runnable entity — process (single thread), thread from multithreaded process, or kernel thread |
| BPF program | Kernel-mode program running in the BPF execution environment |
| Main memory | Physical memory of the system (RAM) |
| Virtual memory | Abstraction of main memory — supports multitasking and oversubscription — practically infinite |
| Kernel space | Virtual memory address space for the kernel |
| User space | Virtual memory address space for processes |
| Context switch | Switch from running one thread/process to another — switches CPU register set (thread context) |
| Mode switch | Switch between kernel and user modes |
| System call (syscall) | Well-defined protocol for user programs to request kernel to perform privileged operations |
| Trap | Signal sent to kernel to request a system routine — types include syscalls, exceptions, and interrupts |
| Hardware interrupt | Signal sent by physical devices to kernel — usually to request servicing of I/O |

## Background

### Kernel

- Monolithic kernel (Linux, BSD) — manages CPU scheduling, memory, file systems, network protocols, system devices
- System libraries — provide richer programming interface than syscalls alone — applications can bypass and call syscalls directly
- BPF — extends kernel model — allows secure kernel-mode programs with their own kernel API (BPF helpers)
- Kernel executes on demand — system calls, device interrupts, and some asynchronous housekeeping threads
- Kernel affects compute-intensive workloads via — CPU contention, CPU selection (cache warmth, memory locality)

### Kernel and User Modes

- Kernel mode — full access to devices and privileged instructions
- User mode — requests privileged operations via system calls
- Implemented using privilege rings — x86 supports 4 rings (0–3) — typically 2 or 3 used
- Mode switch — switching between user and kernel modes
- Context switch — occurs for blocking syscalls (disk/network I/O) — allows another thread to run while first is blocked

#### Avoiding Mode/Context Switch Overhead

| Technique | Description |
|-----------|-------------|
| User-mode syscalls (vDSO) | Kernel exports virtual DSO mapped into process address space — Eg - `gettimeofday(2)`, `getcpu(2)` |
| Memory mappings | Demand paging, data stores, I/O — avoids syscall overhead |
| Kernel bypass | User-mode direct device access — Eg - DPDK for networking |
| Kernel-mode applications | TUX web server, extended BPF programs |

### System Calls

- Hundreds available — kernel maintainers aim to keep count small (Unix philosophy)
- Documented in man page section 2

#### Key System Calls

| Syscall | Description |
|---------|-------------|
| `read(2)` | Read bytes |
| `write(2)` | Write bytes |
| `open(2)` | Open a file |
| `close(2)` | Close a file |
| `fork(2)` | Create a new process |
| `clone(2)` | Create a new process or thread |
| `exec(2)` | Execute a new program |
| `connect(2)` | Connect to a network host |
| `accept(2)` | Accept a network connection |
| `stat(2)` | Fetch file statistics |
| `ioctl(2)` | Set I/O properties or miscellaneous functions |
| `mmap(2)` | Map a file to memory address space |
| `brk(2)` | Extend the heap pointer |
| `futex(2)` | Fast user-space mutex |

- `ioctl(2)` — used for miscellaneous kernel actions where no other syscall is suitable — Eg - `perf_event_open(2)` returns FD then `ioctl(fd, PERF_EVENT_IOC_ENABLE)` enables instrumentation
- `mmap(2)` — maps executables, libraries, memory-mapped files — sometimes used instead of `brk(2)`-based `malloc(2)` to reduce syscall rate
- `brk(2)` — extends heap pointer — called by memory allocator when `malloc(3)` cannot satisfy from existing heap
- `futex(2)` — handles the blocking part of a user-space lock

### Interrupts

- Signal to processor that an event needs processing — interrupts current execution
- Causes processor to enter kernel mode, save thread state, run interrupt service routine (ISR)

#### Asynchronous Interrupts (Hardware)

- Generated by external hardware — arrive asynchronously to currently running software
- Examples — disk I/O completion, hardware failure, network packet arrival, keyboard/mouse input

#### Synchronous Interrupts (Software)

| Type | Description |
|------|-------------|
| Traps | Deliberate kernel call — Eg - `int 0x80` on Linux x86 for syscalls |
| Exceptions | Exceptional condition — Eg - divide by zero |
| Faults | Memory events — Eg - page faults from accessing memory without MMU mapping |

#### Interrupt Processing

- ISRs designed to run as fast as possible — Linux device drivers use two halves —
    - Top half — handles interrupt quickly in interrupt-disabled mode
    - Bottom half — scheduled for later processing — tasklets or work queues (threads that can sleep)
- Network drivers — top half handles IRQs for inbound packets — bottom half (softirq) pushes packet up network stack
- Interrupt latency — time from interrupt arrival to when serviced — critical for real-time systems

#### Interrupt Masking

- Some kernel code paths cannot be safely interrupted (Eg - spin lock held that might be needed by interrupt)
- Kernel temporarily masks interrupts via CPU interrupt mask register
- Interrupt disabled time must be short — perturbs application timely execution
- Non-maskable interrupts (NMIs) — cannot be masked — Eg - IPMI watchdog timer to detect kernel lockup

### Clock and Idle

- `clock()` / `scheduler_tick()` — executed from timer interrupt — historically 60, 100, or 1000 Hz
- Functions — update system time, expire timers/time slices, maintain CPU stats, execute scheduled routines

#### Clock Performance Issues

| Issue | Description | Fix |
|-------|-------------|-----|
| Tick latency | Up to 10 ms additional latency for 100 Hz clock | High-resolution real-time interrupts |
| Tick overhead | Consumes CPU cycles, perturbs applications (OS jitter), prevents CPU power-down | Tickless kernel (NO_HZ) |

- Modern Linux — `NO_HZ` functionality omits clock calls when no CPU load — configured by `CONFIG_NO_HZ`
- Idle thread — runs when no work available — calls `hlt` instruction to power down CPU until next interrupt

### Processes

- Environment for executing user-level program — memory address space, file descriptors, thread stacks, registers
- Identified by PID — multitasked by kernel — thousands concurrent
- Contains one or more threads — threads share address space and file descriptors
- On Linux — threads and processes are both tasks

#### Process Creation

- fork(2) / clone(2) → exec(2)

- Linux C libraries implement `fork` by wrapping `clone(2)`
- Copy-on-write (COW) — adds references to previous address space rather than copying — defers/eliminates memory copying

#### Process Life Cycle

| State | Description |
|-------|-------------|
| on-proc | Running on a CPU |
| ready-to-run | Runnable but waiting on CPU run queue |
| sleep | Blocked on I/O — woken when I/O completes |
| zombie | Terminating — waiting for parent to reap process status |

#### Process Environment

- Kernel context — PID, owner UID, various times, file descriptors
- User address space — executable, libraries, heap, thread stacks
- Each thread has its own user stack and kernel exception stack

### Stacks

- LIFO memory for temporary data — saves/restores return addresses and registers across function calls
- Stack frame — set of data on stack related to one function's execution
- Stack walking — examining saved return addresses across all stack frames — reveals call path
- Stack back trace (stack trace) — call path to currently executing function — invaluable for debugging and performance

#### How to Read a Stack

tcp_sendmsg+1           ← currently executing (leaf)

sock_sendmsg+62         ← caller

SYSC_sendto+319         ← grandparent

sys_sendto+14

do_syscall_64+115

entry_SYSCALL_64_after_hwframe+61  ← root

- Printed leaf-to-root — first line = current function — below = parent, grandparent, etc.
- Read bottom-up to follow execution path — "how we got here"
- Offset to right of function name — instruction location within function

#### User and Kernel Stacks

- Thread executing syscall has two stacks — user-level stack and kernel-level stack
- User stack does not change during syscall — thread uses kernel stack for kernel execution
- Linux also has stacks for softirqs and hard interrupts (IRQs)

### Virtual Memory

- Abstraction of main memory — processes and kernel each get own private, near-infinite address space
- Supports multitasking — each process operates on private address space without contention
- Supports oversubscription — OS transparently maps between main memory and secondary storage

#### Memory Management Schemes

| Scheme | Description |
|--------|-------------|
| Process swapping | Moves entire processes to/from secondary storage — original Unix — severe performance loss |
| Paging | Moves small pages (Eg - 4 KB) between main memory and storage — more efficient — BSD addition |

- Linux term "swapping" refers to paging — does not support Unix-style full process swapping

### Schedulers

- Maps threads (tasks) to CPUs — divides CPU time among active processes and threads
- Maintains priority — higher-priority work executes sooner
- Traditionally per-priority run queues — modern kernels use per-CPU queues and other data structures

#### Workload Categories

| Type | Description |
|------|-------------|
| CPU-bound | Heavy compute — long runtimes — limited by CPU resources |
| I/O-bound | Performs I/O with little compute — limited by storage or network — low-latency responses desirable |

- Classic scheduling policy — identifies CPU-bound workloads (high compute/real-time ratio) and decreases their priority — gives preference to I/O-bound and shorter-running (interactive) processes

### File Systems

- Organisation of data as files and directories — POSIX-based interface
- Global file namespace — top-down tree from root `/` — file systems join via mounting at mount points
- VFS — kernel interface abstracting multiple file system types — allows transparent access

#### I/O Stack

- Path from user-level software to storage device — subset of entire software stack
- Includes direct path to block devices (bypassing file system) — used by admin tools and databases

### Networking

- Built-in network protocol stack (TCP/IP stack)
- User-level applications access network via sockets
- Physical device — network interface card (NIC) — associated with IP address via DHCP
- Growing adoption of QUIC — a new transport protocol

### Device Drivers

- Kernel software for device management and I/O — often provided by hardware vendors
- Pluggable drivers — loaded and unloaded without system restart
- Character devices (raw devices) — unbuffered sequential access (Eg - keyboards, serial ports)
- Block devices — I/O in 512-byte blocks — accessed randomly by block offset — Linux buffer cache now part of page cache

### Multiprocessor

- SMP (Symmetric Multiprocessing) — all CPUs treated equally
- NUMA (Non-Uniform Memory Access) — banks of main memory connected to different sockets — performance challenges
- IPI (Inter-Processor Interrupt) — CPU requests other CPUs to immediately perform work (Eg - cache coherency of TLB entries) — also used by preemption

### Preemption

- Allows high-priority user-level threads to interrupt the kernel — required for real-time systems
- Fully preemptible kernel — allows interruption at almost any point
- Voluntary kernel preemption (`CONFIG_PREEMPT_VOLUNTARY`) — logical stopping points check and perform preemption — lower complexity, low latency for common workloads
- `CONFIG_PREEMPT` — all kernel code except critical sections preemptible
- `CONFIG_PREEMPT_NONE` — no preemption — highest throughput, highest latency

### Resource Management

- Controls for fine-tuning access to CPUs, memory, disk, and network
- Early Unix/BSD — `nice(1)` for CPU priorities, `ulimit(1)` for resource limits
- Linux cgroups (v1, `2.6.24`) — measure and limit resource usage for groups of processes
- cgroup v2 (`4.5`) — unified hierarchical scheme — stable and exposed in Linux 4.5
- Documented in kernel source under `Documentation/cgroups` and `Documentation/admin-guide/cgroup-v2.rst`

### Unix

- Developed at AT&T Bell Labs starting 1969 — multitasked, lightweight
- Performance features — process scheduler priorities, 512-byte block disk I/O with buffer cache, idle process swapping, multitasking

### BSD

- Began as Unix 6th Edition enhancements at UC Berkeley — first released 1978
- Rewritten under BSD license by early 1990s — enabled FreeBSD

#### Key BSD Performance Developments

| Feature | Description |
|---------|-------------|
| Paged virtual memory | Moved smaller LRU memory chunks instead of entire processes |
| Demand paging | Defers physical memory mapping until first write |
| FFS | Berkeley Fast File System — cylinder groups — reduced fragmentation |
| TCP/IP network stack | First high-performance TCP/IP stack for Unix (4.2BSD, 1983) |
| Sockets | Berkeley sockets API — 4.2BSD — became the networking standard |
| Jails | Lightweight OS-level virtualisation — FreeBSD 4.0 |
| Kernel TLS | Moves TLS processing to kernel — improves performance |

### Solaris

- Unix/BSD-derived kernel — Sun Microsystems 1982 — based on SVR4

#### Key Solaris Performance Developments

| Feature | Description |
|---------|-------------|
| VFS | Virtual file system — abstraction for multiple file system types |
| Fully preemptible kernel | Low latency for high-priority and real-time work |
| Multiprocessor support | ASMP and SMP support in early 1990s |
| Slab allocator | Per-CPU caches of preallocated buffers — replaced buddy allocator — became kernel standard |
| DTrace | Static and dynamic tracing framework — entire software stack visibility in production |
| Zones | OS-based virtualisation — shared kernel — influenced Linux containers |
| ZFS | Enterprise file system — available for Linux via OpenZFS |

- Inherited from — Unix/Multics (syscalls, multitasking, virtual memory, VFS), BSD (paged VM, FFS, TCP/IP), Solaris (VFS, page cache, slab allocator), Plan 9 (rfork)

### systemd

- Service manager — replacement for original Unix init system
- Dependency-aware service startup — service time statistics

```bash
systemd-analyze                  # overall boot time
systemd-analyze critical-chain   # critical path and slowest services
systemd-analyze blame            # slowest initialisation times
systemd-analyze plot             # SVG diagram of boot
```

### KPTI (Meltdown)

- Kernel page table isolation — added Linux 4.14 — mitigates Intel Meltdown vulnerability
- Reduces processor performance — extra CPU cycles and TLB flushing on context switches and syscalls
- PCID support (same release) — avoids some TLB flushes if processor supports it
- Performance impact at Netflix — 0.1%–6% depending on syscall rate
- Mitigation — huge pages (faster TLB warmup), tracing to reduce syscall rate

### Extended BPF

- General-purpose execution engine — in-kernel — virtual machine with its own instruction set
- Runs in kernel mode — configured to run on events

#### BPF Components

| Component | Description |
|-----------|-------------|
| Instruction set | Virtual machine instruction set — compiled to native via JIT |
| Maps | Storage objects — suited for statistics and key-value data |
| Helper functions | Kernel API for BPF programs |
| Verifier | Checks program safety before loading — ensures no crash or kernel corruption |
| BTF (BPF Type Format) | System for understanding data types and structures |

#### BPF Event Sources

- Socket events
- Tracepoints
- USDT probes
- kprobes
- uprobes
- perf_events

#### BPF Output Mechanisms

- perf ring buffer — efficient per-event data emission
- Maps — suited for statistics and histograms

#### Key BPF History

| Version | Feature |
|---------|---------|
| 3.19 | Attached to kprobes |
| 4.7 | Attached to tracepoints |
| 4.9 | Attached to software and hardware events |
| 4.10 | Attached to cgroups |
| 5.3 | Bounded loops — increased instruction limit for complex applications |

- Front-ends — BCC and bpftrace (covered in Chapter 15)
- Enables new generation of efficient, safe, advanced tracing tools
- Powers Netflix FlameCommander, tcpretrans, biolatency, offcputime, and hundreds of other tools

## Other Kernel Models

| Model | Description | Examples |
|-------|-------------|---------|
| Monolithic | All device management code in single large kernel program | Linux, BSD, Solaris |
| Microkernel | Minimal kernel — file systems, network stack, drivers as user-mode software | QNX, Minix 3 |
| Hybrid kernel | Combines monolithic and microkernel — performance-critical services in kernel space, modular design | Windows NT, Plan 9 |
| Unikernel | Single-application machine image — kernel + library + application combined — single address space | MirageOS |
| Distributed OS | Single OS instance across multiple networked nodes | Plan 9, Inferno |
| PGO kernel | Profile-guided optimisation — recompile kernel based on CPU profile from production | Used by Google and Microsoft |

> [!NOTE]
> Unikernels present an observability challenge — no SSH, shells, or performance tools available by default. New performance tooling must be built specifically for them.

> [!TIP]
> Linux is the dominant choice for performance-critical production environments — 100% of TOP500 supercomputers use Linux since 2017. However, Netflix uses FreeBSD for its CDN due to specific workload optimisations made by the Netflix OCA team. Kernel performance comparisons require expert analysis — micro-benchmarks are error-prone and may not reflect production workloads.
